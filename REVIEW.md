# Проблемки

## 1. Гонка данных при доступе к складу `s.apples`

Поле `s.apples` может меняться сразу в нескольких местах.

1) Из горутины разгрузки (`go func()` внутри `BeginUnloading`, строка `s.apples = append(s.apples, box)`)
2) Из горутины клиента (`PlaceOrderSimple`: `sort.Slice(s.apples, ...)` и `s.apples[i].Quantity -= selected`)
3) Из `revertReservation`.

Никакой синхронизации (`sync.Mutex` / `sync.RWMutex` / канала) нет.

Если поставщик будет разгружаться одновременно с заказом покупателя, то во время реаллацирования может случиться обращение к инвалидному индексу

---

## 2. Гонка при работе с парковкой `freeChan` и `idToChan`

работа с `s.freeChan` (в `BeginUnloading` и `FinishUnloading`) ведется без мьютекса, что очень опасно.

Представим, что два поставщика одновременно вызывают `BeginUnloading`. Оба видят `len(s.freeChan) == 1`, оба берут «последний» канал. один поставщик получает занятый канал второго, второй паникует на индексе `-1`. Параллельный вызов `FinishUnloading` другого поставщика приводит к падению рантайма на map.

---

## 3. Busy-wait в `BeginUnloading`

Ожидание свободного места на парковке реализовано как `for len(s.freeChan) == 0 { time.Sleep(13 * time.Millisecond) }`. Это активное ожидание + гонка чтения `len`.
Если будет много поставщиков и много горутин крутятся в коротких циклах сна, то будет жесткий+мощный расход CPU

---

## 4. Гонка на `nextOrderId` и неуникальные ID заказов

`startDelivery` читает `s.nextOrderId`, использует его в качестве ID, а потом ещё делает `s.nextOrderId++` три операции без атомарности и без мьютекса.

Представим, что клиента одновременно вызывают `PlaceOrderSimple`. Оба читают `nextOrderId == 5`, оба возвращают клиенту ID 5, оба увеличивают счётчик до 6 (а не до 7). При том, что в задании ID должен быть глобально уникальным.
