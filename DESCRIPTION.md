### Поставщики

Яблоки хранятся на центральном складе, куда их привозят поставщики на своих грузовиках. Каждый поставщик представлен своей собственной горутиной (наш сервис не управляет ей), которая взаимодействует с нашим сервисом с помощью вызова функций из интерфейса [SupplierApi](src/api/supplier.go). 

Количество парковочных мест у склада ограничено, поэтому поставщик может ждать, пока освободится парковочное место. Когда же поставщик смог припарковаться, он начинает разгрузку своих яблок на склад и информирует нас об этом путем отправки сообщений в канал, полученный им из 
`BeginUnloading`

Логикой того, как именно поставщик будет разгружаться и отправлять в канал сообщения наш сервис не управляет — он может разгрузить все яблоки и отправить сообщение разом, а может относить по одному яблоку за раз (и каждый раз отправлять сообщение) с большими временным перерывами между отправками

Когда поставщик полностью заканчивает разгрузку грузовика, он уезжает с парковки и вызывает функцию `FinishUnloading`, чтобы сообщить нашему сервису, что он выполнил поставку

```aiignore
Схема (размер парковки 2):
Горутина поставщика A: BeginUnloading - send -- send - FinishUnloading
Горутина поставщика B: BeginUnloading ----------- send ------------ FinishUnloading
Горутина поставщика C: (wait for parking) ------------ BeginUnloading - send - FinishUnloading
```

### Потребители

Клиенты-потребители заказывают яблоки через мобильное приложение. Каждый покупатель представлен своей горутиной (наш сервис не управляет ей), которая взаимодействует с нашим сервисом с помощью вызова функций из интерфейса [ConsumerApi](src/api/consumer.go)

Есть три типа заказа: 
- `SimpleOrder`: заказ на яблоки одного конкретного сорта
- `AnyApplesOrder`: заказ на яблоки, сорта яблок неважны
- `MultiOrder`: заказ с конкретными сортами яблок, для каждого сорта указано свое количество

Потребителям очень важна свежесть яблок, поэтому при заказе клиент указывает минимально допустимую дату годности. Все яблоки, выбранные под заказ, должны иметь срок годности не раньше этой даты.

Потребители очень требовательны, им нужно доставить **все** яблоки из заказа (т.е. частичное исполнение недопустимо)

Если заказ пользователя не может быть удовлетворен (например потому что у нас нет достаточного числа нужных яблок с подходящим сроком годности), то вызов функции `PlaceOrderSimple`/`PlaceOrderMulti`/`PlaceOrderAny` должен вернуть ошибку

Иначе наш сервис подтверждает заказ, асинхронно запускает доставку в отдельной горутине, возвращает глобальный уникальный идентификатор заказа

После заказа клиент может через приложение в любой момент (если заказ еще не получен) отменить свой заказ с помощью функции `Cancel`, куда должен передать свой идентефикатор заказа. В рамках проекта считаем, что клиенты будут отменять только свои заказы (в реальной жизни это будет проверять внешний сервис аутентификации и авторизации, который не связан напрямую с нашим сервисом)

### Доставка

После подтверждения заказа потребителя заказ должен быть доставлен. Для этого уже существует внешний сервис доставки, с которым наш сервис должен взаимодействовать: [DeliveryService](src/external_services/delivery.go)

Этот сервис содержит только один метод: 
```go
DoDelivery(ctx context.Context, req DeliveryRequest) error
```

Когда наш сервис вызывает эту функцию — курьер возьмет со склада яблоки клиента и пойдет доставлять их. Это блокирующая функция (так уж написали этот сервис доставки), т.е. мы выйдем из этой функции, только когда курьер *полностью* выполнит все вышеописанные действия. Поэтому её вызов может занимать существенно время по длительности, из-за чего функция поддерживает отмену через контекст.

Если курьер доставил заказ, то `DoDelivery` возвращает `nil`

Если заказ отменён клиентом до завершения `DoDelivery`, наш сервис должен попытаться отменить доставку через `context.Context`

Если курьер не смог доставить заказ (например его отменил клиент, курьер не смог вручить заказ, или произошла какая-то другая ошибка), то `DoDelivery` возвращает ошибку

Если `DoDelivery` завершился с ошибкой, все яблоки, ранее зарезервированные под заказ, должны снова стать доступными на складе, если их срок годности к этому моменту ещё не истёк (курьеры возвращают в таком случае яблоки на склад и мы должны учесть это в нашем сервисе)

### Компостер

К сожалению, не все яблоки успевают раскупить до истечения их срока годности. В этом случае наш сервис должен отправлять их на переработку в удобрения с помощью [компостера](https://www.youtube.com/watch?v=JJzaEZNlGbU).

Для этого уже существует внешний сервис компостирования, с которым наш сервис должен взаимодействовать: [ComposterService](src/external_services/composter.go)
 
Работа с компостером требует соблюдения техники безопасности: в него можно класть любые яблоки, но только если он выключен, а его крышка открыта. Яблоки надо класть по одному, иначе при запуске компостер сломается. После того как яблоки положены в компостер (стоит учесть, что это делается через буферизированный канал ограниченного объема), надо закрыть защитную крышку компостера и включить его. Компостер переработает яблоки, после этого его надо выключить.

Все яблоки, срок годности которых истёк, должны быть удалены со склада и переданы в компостер фоновой горутиной.

Для упрощения считаем, что зарезервированные и переданные курьеру яблоки не утилизируются компостером. Фоновая утилизация работает только с яблоками, лежащими на складе

Для получения текущего времени используйте функцию `time.Now()`
